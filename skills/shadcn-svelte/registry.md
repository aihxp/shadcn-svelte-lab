# Registry Authoring and Addresses

Use this reference when the user wants to create, fix, publish, inspect, or reason about a shadcn-svelte registry.

## Mental Model

A registry has two forms:

- Source registry: an authored `registry.json` that points at source files.
- Built registry: generated JSON files served to CLI consumers, usually from `static/r` or another public directory.

The installer consumes built registry item JSON. The source registry is the authoring format that `shadcn-svelte registry build` turns into those payloads.

Registry items can distribute Svelte components, blocks, hooks, utilities, CSS variables, fonts, pages, docs, agent rules, templates, and other project files.

## Root `registry.json`

The root registry file must define registry metadata and `items`.

```json
{
  "$schema": "https://shadcn-svelte.com/schema/registry.json",
  "name": "acme",
  "homepage": "https://acme.com",
  "items": [
    {
      "name": "hello-world",
      "type": "registry:block",
      "title": "Hello World",
      "description": "A simple Svelte block.",
      "files": [
        {
          "path": "src/lib/registry/blocks/hello-world.svelte",
          "type": "registry:component"
        }
      ],
      "registryDependencies": ["button"]
    }
  ]
}
```

Root registry rules:

- `name` and `homepage` are required.
- `items` is an array of registry item definitions.
- Local `registry build` currently reads the root file's `items`.
- `include` is supported for GitHub source registries consumed directly by address, but local build output should use a flat root `items` list until local include build support lands.

## Item Definitions

Common item fields:

```json
{
  "name": "login-form",
  "type": "registry:block",
  "title": "Login Form",
  "description": "A login form with email and password fields.",
  "dependencies": ["zod"],
  "registryDependencies": ["button", "input", "field"],
  "files": [
    {
      "path": "src/lib/registry/blocks/login-form.svelte",
      "type": "registry:block"
    }
  ],
  "cssVars": {
    "theme": {
      "brand": "oklch(0.62 0.18 250)"
    },
    "light": {
      "brand": "oklch(0.62 0.18 250)"
    },
    "dark": {
      "brand": "oklch(0.72 0.16 250)"
    }
  }
}
```

Important fields:

- `name`: the installable item name. It is not necessarily a file path.
- `type`: one of `registry:ui`, `registry:block`, `registry:component`, `registry:lib`, `registry:hook`, `registry:file`, `registry:page`, `registry:theme`, `registry:style`, `registry:font`, or `registry:item`.
- `files`: source files copied or generated by the item.
- `dependencies`: npm runtime dependencies.
- `devDependencies`: npm development dependencies.
- `registryDependencies`: other registry items required by this item.
- `cssVars`, `css`, `font`, and `docs`: optional install-time additions.

File rules:

- File paths in source registries are relative to the declaring `registry.json`.
- Built item files use `target` to decide where files install.
- `registry:file` and `registry:page` files should include an explicit `target`.
- Keep registry files copyable outside your app: avoid hidden app-only imports.

## Aliases

Use `aliases` when your registry source imports from internal paths. The build command replaces those source aliases with placeholders so installs can resolve to the consumer project's aliases.

```json
{
  "aliases": {
    "lib": "$lib/registry/lib",
    "ui": "$lib/registry/ui",
    "components": "$lib/registry/components",
    "utils": "$lib/utils",
    "hooks": "$lib/registry/hooks"
  }
}
```

For Svelte registries, prefer these import shapes in source files:

```svelte
<script lang="ts">
  import { Button } from "$lib/registry/ui/button";
  import * as Field from "$lib/registry/ui/field";
</script>
```

## Registry Dependencies

`registryDependencies` entries are item addresses, not file paths.

```json
{
  "name": "login-form",
  "type": "registry:block",
  "registryDependencies": ["button", "@acme/input", "acme/ui/card#v1.2.0"]
}
```

Dependency rules:

- Bare names such as `"button"` mean official shadcn-svelte items.
- Namespaced dependencies use `@namespace/item-name`.
- GitHub dependencies use `owner/repo/item-name`.
- Pin GitHub dependencies with `owner/repo/item-name#ref` when needed.
- Refs are not inherited. If `owner/repo/foo#v2` depends on `bar` from the same repo at `v2`, write `owner/repo/bar#v2`.

## Address Schemes

Classify item strings before fetching or installing them.

| Address                             | Scheme    | Meaning                                               |
| ----------------------------------- | --------- | ----------------------------------------------------- |
| `button`                            | official  | Official shadcn-svelte item named `button`.           |
| `@acme/button`                      | namespace | Item `button` from configured registry `@acme`.       |
| `@acme/ui/button`                   | namespace | Item `ui/button` from configured registry `@acme`.    |
| `https://example.com/r/button.json` | url       | Built registry item JSON at that URL.                 |
| `./button.json`                     | file      | Built registry item JSON on disk.                     |
| `acme/ui/button`                    | github    | Item `button` from GitHub repo `acme/ui`.             |
| `acme/ui/forms/login#main`          | github    | Item `forms/login` from GitHub repo `acme/ui` at ref. |

For namespace and GitHub addresses, slashful item names are allowed and are item names, not file paths.

## GitHub Registries

A public GitHub repository can act as a source registry when it has a root `registry.json`.

```txt
owner/repo/item-name[#ref]
```

Rules:

- The first two path segments are GitHub owner and repo.
- All remaining path segments are the registry item name.
- The source entrypoint is always root `registry.json`.
- `include` paths must be relative paths inside the same repository and must point to a `registry.json` file.
- GitHub registries are consumed directly by the CLI. They do not require generated item JSON files.
- Current support is for public `github.com` repositories.

## Build and Verify

Build a local source registry:

```bash
npx shadcn-svelte@latest registry build
npx shadcn-svelte@latest registry build registry.json --output static/r
```

Inspect built or remote items:

```bash
npx shadcn-svelte@latest search @acme -q "login"
npx shadcn-svelte@latest view @acme/login-form
npx shadcn-svelte@latest docs button
npx shadcn-svelte@latest add @acme/login-form
```

Validate a public GitHub source registry:

```bash
npx shadcn-svelte@latest registry validate owner/repo
```

Implementation guidance:

- Keep address parsing pure and testable.
- Preserve official, namespace, URL, file, and GitHub address behavior.
- Add tests for address parsing, source loading, dependency resolution, search, view, and add paths.
- Resolve GitHub refs before reading source files so one command uses one consistent repository snapshot.
